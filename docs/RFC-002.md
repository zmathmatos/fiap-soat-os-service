# RFC-002 — Autenticação de Customers via AWS API Gateway + Lambda Authorizer

**Status:** Aceito  
**Data:** 2026-05-24  
**Autor:** Time de Engenharia  
**Relacionado:** RFC-001 (Escolha da AWS)

---

## Contexto

As rotas de customer (`/customer/service-orders/*`) permitem que um usuário consulte e interaja com ordens de serviço usando apenas o número do documento (CPF/CNPJ) como identificador. Sem um mecanismo de autenticação, qualquer requisição pode:

1. Acessar as ordens de serviço sem ter cadastro no sistema.
2. Aprovar ou rejeitar cotações sem ter cadastro no sistema.

O backend roda em Express.js dentro de um cluster EKS e possui um mecanismo de autenticação próprio para o fluxo de customers, igual do fluxo administrativo que usa JWT + `authMiddleware`.

## Decisão

Introduzir um **AWS API Gateway** na frente das rotas de customer, com uma **AWS Lambda Authorizer** responsável por autenticar cada requisição antes de ela chegar ao backend.

**Repositório da Lambda:** `fiap-soat-tech-challenge-lambda`

---

## Fluxo de Autenticação

```
Cliente
  │
  │  GET /customer/service-orders/...
  │  Header: x-document: <CPF ou CNPJ>
  ▼
API Gateway
  │
  │  Chama Lambda Authorizer (síncronamente)
  ▼
Lambda Authorizer
  │  1. Extrai header x-document
  │  2. Valida formato do documento (CPF/CNPJ)
  │  3. Consulta RDS: SELECT * FROM users WHERE document = ?
  │  4a. Usuário NÃO encontrado → retorna policy DENY → API GW responde 403
  │  4b. Usuário encontrado → gera JWT com { userId, document, role }
  │                        → retorna policy ALLOW + JWT no context
  ▼
Backend EKS (Express.js)
  │  Recebe a requisição com o JWT injetado pelo API Gateway
  │  Valida o token
  ▼
RDS PostgreSQL
```

---

## Justificativa

### Separação de responsabilidades

A lógica de autenticação fica isolada na Lambda, fora do domínio da aplicação principal. O backend Express só precisa confiar no JWT já validado que chega via API Gateway, sem acoplamento com a lógica de verificação de documento.

### Proteção contra acesso não autorizado

O header `x-document` vincula cada requisição a um usuário real no banco. Sem cadastro, nenhuma requisição chega ao backend — o bloqueio ocorre na borda, antes de consumir recursos da aplicação.

### Custo e escala

Evita que requisições de usuário não autorizados cheguem até nossa aplicação. Isso reduz as chamadas ao nosso backend, mantendo custo operacional baixo.

### Consistência com a infraestrutura AWS existente

Utiliza serviços da AWS que já estava previsto na arquitetura (RFC-001). Utilizar uma Lambda Authorizer é o padrão nativo da AWS para esse caso de uso, sem introduzir dependências externas.

## Rotas Protegidas

| Método | Rota | Proteção |
|--------|------|----------|
| `GET` | `/customer/service-orders/number/:serviceOrderNumber` | Lambda Authorizer |
| `GET` | `/customer/service-orders/user/document/:document` | Lambda Authorizer |
| `GET` | `/customer/service-orders/vehicle/license-plate/:licensePlate` | Lambda Authorizer |
| `GET` | `/customer/service-orders/:serviceOrderNumber/quotation/approval` | Lambda Authorizer |
| `GET` | `/customer/service-orders/:serviceOrderNumber/quotation/rejection` | Lambda Authorizer |

---

## Consequências

- O backend deve validar o JWT recebido e usar o `userId` do payload para filtrar ordens de serviço por usuário.
- A Lambda precisa de acesso de leitura ao RDS via VPC (security group dedicado).
- O segredo JWT compartilhado entre a Lambda e o backend deve ser armazenado no action secrets do github.
